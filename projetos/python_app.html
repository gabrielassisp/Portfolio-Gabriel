Sales Dashboard ETL

Pipeline ETL completo que extrai dados de uma API pública de produtos, transforma, persiste em banco de dados SQLite via SQLAlchemy ORM e exibe um dashboard interativo com Streamlit — com atualização automática a cada 60 minutos via job agendado. O projeto segue boas práticas de engenharia de software: módulos separados, testes unitários com pytest e variáveis de ambiente com dotenv.

# etl.py — Extração, transformação e carga dos dados de vendas import logging, requests import pandas as pd from src.database import SessionLocal, engine from src.models import Base, Produto, Venda   logger = logging.getLogger(__name__)   def extrair_produtos() -> list[dict]:     """Busca produtos da Fake Store API."""     resp = requests.get("https://fakestoreapi.com/products", timeout=10)     resp.raise_for_status()     return resp.json()   def transformar_produtos(dados: list[dict]) -> pd.DataFrame:     """Limpa e normaliza os dados brutos da API."""     df = pd.DataFrame(dados).rename(columns={"title":"nome","price":"preco","category":"categoria"})     df["nota"] = df["rating"].apply(lambda x: x["rate"])     df["preco"] = df["preco"].round(2)     return df[["id","nome","preco","categoria","nota"]]   def rodar_etl() -> None:     """Ponto de entrada do pipeline ETL."""     logger.info("=== Iniciando ETL ===")     dados = extrair_produtos()     df_produtos = transformar_produtos(dados)     carregar_banco(df_produtos, gerar_vendas_simuladas(df_produtos["id"].tolist()))     logger.info("=== ETL concluído ===")
# queries.py — Consultas analíticas centralizadas import pandas as pd from sqlalchemy import text from src.database import engine   def receita_por_dia(dias: int = 30) -> pd.DataFrame:     """Receita total e unidades vendidas por dia."""     sql = text("""     SELECT data_venda,            SUM(receita) AS receita_total,            SUM(quantidade) AS unidades_vendidas     FROM vendas     WHERE data_venda >= DATE('now', :offset)     GROUP BY data_venda ORDER BY data_venda     """)     with engine.connect() as conn:         return pd.read_sql(sql, conn, params={"offset": f"-{dias} days"})   def top_produtos(n: int = 10) -> pd.DataFrame:     """Top N produtos por receita gerada."""     sql = text("""     SELECT p.nome, p.categoria, SUM(v.receita) AS receita_total,            SUM(v.quantidade) AS unidades_vendidas     FROM vendas v JOIN produtos p ON p.id = v.produto_id     GROUP BY p.id ORDER BY receita_total DESC LIMIT :n     """)     with engine.connect() as conn:         return pd.read_sql(sql, conn, params={"n": n})
# models.py — Definição das tabelas via SQLAlchemy ORM from sqlalchemy import Column, Integer, Float, String, Date, ForeignKey from sqlalchemy.orm import declarative_base, relationship   Base = declarative_base()   class Produto(Base):     """Tabela de produtos do catálogo."""     __tablename__ = "produtos"     id = Column(Integer, primary_key=True)     nome = Column(String(300), nullable=False)     preco = Column(Float, nullable=False)     categoria = Column(String(100), nullable=False)     nota = Column(Float)     vendas = relationship("Venda", back_populates="produto")   class Venda(Base):     """Tabela de registros de vendas."""     __tablename__ = "vendas"     id = Column(Integer, primary_key=True, autoincrement=True)     produto_id = Column(Integer, ForeignKey("produtos.id"), nullable=False)     quantidade = Column(Integer, nullable=False)     receita = Column(Float, nullable=False)     data_venda = Column(Date, nullable=False)     produto = relationship("Produto", back_populates="vendas")

Contexto do Projeto
Projeto de portfólio focado em demonstrar um pipeline de dados completo do ponto de extração até a visualização — com a mesma estrutura que você encontra em times de dados reais. A fonte é a Fake Store API (pública), com vendas simuladas com sazonalidade de fim de semana.
Boas Práticas de Desenvolvimento
  • Código organizado em módulos (src/), não um único notebook
  • Testes unitários com pytest sem dependência de banco ou internet
  • Credenciais via .env — nunca hardcoded no código
  • ORM em vez de SQL cru — modelo fácil de migrar para Postgres
Decisões Técnicas
  • SQLite elimina necessidade de servidor externo para rodar local
  • schedule em vez de cron — portável em Windows e Linux
  • Queries separadas da UI facilitam testes e reuso
  • @st.cache_data evita queries redundantes no Streamlit
